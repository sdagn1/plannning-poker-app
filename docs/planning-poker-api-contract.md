# Planning Poker — Backend API Contract

## Overview

The backend is a single WebSocket endpoint plus one trivial HTTP health-check route. Rooms are created implicitly: the first client to connect with a given `room` ID becomes that room's **host**; all later clients connecting with the same `room` ID join as regular users.

Room IDs are generated by the frontend (e.g. a random slug like `xk29-4plm`). How the room ID is surfaced in the URL (query param, hash fragment, path with SPA rewrites, etc.) is entirely up to the frontend team — the backend has no opinion on it.

The backend runs as a single Node.js process with all room state held in memory. There is no persistence, no reconnect/resume, and no shared/clustered state.

Live backend url: https://planning-poker-backend-ymq7.onrender.com

## Connecting

```
wss://<host>/ws?room=<roomId>&name=<displayName>
```

| Param  | Required | Description |
|---|---|---|
| `room` | yes | Room identifier, generated by the client. Must match `^[a-z0-9]{4,32}(-[a-z0-9]{4,32})*$` (lowercase alphanumeric, segments 4–32 chars separated by single dashes, total length ≤ 128). If the room doesn't exist yet, it's created and this connection becomes host. |
| `name` | yes | Display name shown to other participants. Trimmed; 1–32 chars after trim. |

If `room` or `name` fails validation, the server rejects the HTTP upgrade with `400 Bad Request` — the WebSocket never opens. If the room is already at capacity (`MAX_ROOM_USERS` (default 50) connections), the upgrade is rejected with `403 Forbidden`.

Each WebSocket connection is treated as a fresh session — there is no reconnect/resume support. If a client disconnects and reconnects, it joins as a new user (a new `id` is assigned, a new `seq` is assigned at the back of the line, and any previous vote of the old session is discarded with the old user).

## HTTP routes

### `GET /health`

Returns `200 OK` with body `ok`. Stateless. Used by load balancers and for keep-warm pings to prevent the hosting provider from spinning the service down during a workshop.

No CORS headers are sent; the route is intended for direct browser/probe access, not cross-origin fetch.

## Server → Client messages

### `welcome`

Sent once to a client, immediately after its connection is accepted — before any `state` broadcast.

```json
{ "type": "welcome", "userId": "user-abc123" }
```

Tells the client its own assigned `id`. The frontend should store this and compare it against `state.users[].id` and `state.hostId` to know which user is "me" and whether it's currently host. This is informational only — clients never send `userId` back to the server; the server always identifies the sender by its own connection-to-user mapping, so there's no field a client could spoof to act as another user.

### `state`

Broadcast to **all** clients in a room whenever anything changes (join, leave, vote, reveal, reset, host change). The broadcast always includes the originator. Every `state` message is a full snapshot — clients should replace, not merge.

```json
{
  "type": "state",
  "hostId": "user-abc123",
  "revealed": false,
  "users": [
    { "id": "user-abc123", "name": "Alice", "hasVoted": true },
    { "id": "user-def456", "name": "Bob", "hasVoted": false }
  ],
  "votes": null
}
```

- `votes` is `null` while `revealed` is `false`, so vote values are never leaked early.
- Once revealed, `votes` is populated: `{ "user-abc123": "5", "user-def456": "8" }`.
- `hasVoted` lets the UI show "voted" indicators without exposing values pre-reveal.
- A user disconnecting mid-round is removed from `users[]` and from the `votes` map; no auto-reveal flip occurs. If a host disconnects, the roster change and the host-promotion ride the same `state` message.
- Reconnects are fresh users (new `id`, new `seq`, no vote) and do not regain host, even if they were the original host.

### `error`

Sent only to the client that triggered an invalid action. State is not changed; no `state` broadcast is sent for an `error`.

```json
{ "type": "error", "message": "Only the host can reveal votes" }
```

The full error vocabulary:

| Trigger | Message |
|---|---|
| JSON parse failure | `Malformed message` |
| Missing or unknown `type` | `Unknown message type` |
| Known `type`, bad shape (e.g. `vote` without `value`, `vote.value` non-string) | `Invalid <type> message` |
| `vote` while `revealed === true` | `Voting is locked until reset` |
| `reveal` by non-host | `Only the host can reveal votes` |
| `reset` by non-host | `Only the host can reset votes` |
| `reveal` when already revealed | `Votes are already revealed` |
| `reset` when already reset (no votes, unrevealed) | `Votes are already reset` |
| Inbound rate limit exceeded | `Rate limit exceeded` |
| Unexpected handler exception | `Internal error` |
| Heartbeat pong timeout | (no app-level `error`; socket closed with code 1011) |
| Server shutdown | (no app-level `error`; socket closed with code 1001 "server_shutdown") |

The socket is kept open after any `error` except where noted.

## Client → Server messages

### `vote`

```json
{ "type": "vote", "value": "5" }
```

`value` is a free-form string (so any deck — Fibonacci, T-shirt sizes, "?", "☕", etc. — works without backend changes). Bounds: 1–64 chars after trim, no character restrictions. Sending `vote` with a numeric value (e.g. `5` instead of `"5"`) is rejected with `Invalid vote message`. Re-voting before reveal overwrites the previous vote. `vote` after `reveal` is rejected with `Voting is locked until reset`.

### `reveal`

```json
{ "type": "reveal" }
```

Host-only. Sets `revealed: true` and populates `votes` in the next `state` broadcast. If sent by a non-host, server responds with `Only the host can reveal votes` and ignores the action. If sent when `revealed === true`, server responds with `Votes are already revealed` and ignores the action. Revealing with zero votes is allowed; `votes` will be `{}`.

### `reset`

```json
{ "type": "reset" }
```

Host-only. Clears all votes, sets `revealed: false`, and broadcasts `state` (always, even if already unrevealed with no votes — for predictability the invalid case is rejected instead, see `error` table above). If sent by a non-host, server responds with `Only the host can reset votes` and ignores the action.

### Disconnection

There is **no** `leave` message — closing the WebSocket is the only way to leave. The server's `onClose` handler removes the user from the room, discards their vote, and (if the disconnecting user was host) promotes the remaining user with the smallest `seq` to host, then broadcasts `state`. If the room becomes empty it is discarded immediately.

## Host behavior

- The host is whoever first connects to a given room.
- Each connection is assigned a monotonic per-room `seq` (1, 2, 3, ...) at join time. "Oldest" means smallest remaining `seq`.
- If the host disconnects, the server automatically promotes the remaining user with the smallest `seq` to host, and broadcasts an updated `state` with the new `hostId`. There are no ties (every `seq` is unique within the room).
- If the room becomes empty, it is discarded immediately (no persistence between sessions, no grace period).
- A returning user (reconnect) is treated as a brand-new user with a new `id`, a new `seq` at the back of the line, and no vote. They do **not** regain host even if they were the original host.

## Notes for frontend implementation

- Hide/disable reveal & reset controls for non-host users, but treat the server's `error` response as the source of truth (UI state alone isn't a security boundary).
- Use `hostId` from `state` to determine which connected user is currently host, since it can change at runtime.
- No authentication — anyone with the room URL can join. Treat room IDs as shareable-but-unguessable join codes, not secrets.
- Implement a WS-layer reconnect strategy; the contract guarantees `welcome` → `state` ordering on every new connection, so reconnect logic is the same as initial-join logic.
- Treat server-initiated close codes:
  - `1001 "server_shutdown"` — backend is restarting; reconnect after a short delay (hosting provider may spin down).
- Display names are not unique within a room. Disambiguate in the UI by `id` (or a suffix) if needed.
- The `value` string is returned to the client verbatim (no trimming by the server). Normalize in the UI if desired.

## Operational notes (for the backend team)

- Signal handling: `SIGTERM` triggers graceful shutdown — stop accepting new upgrades, iterate all rooms, send each socket a close frame with code `1001` and reason `"server_shutdown"`, then exit. Bail at `SHUTDOWN_GRACE_MS` (default 20s) if any sockets haven't closed.
- Heartbeat: server-side ping/pong via `ws`, ping every `HEARTBEAT_INTERVAL_MS` (default 30s), terminate any socket that hasn't ponged within `HEARTBEAT_TIMEOUT_MS` (default 10s). Terminated sockets run the normal `onClose` cleanup path.
- Idle rooms are discarded eagerly on the last disconnection; no sweep is needed. The heartbeat loop may perform a defensive empty-room sweep as defense in depth, but `onClose` is the authoritative path.
- No CORS middleware is applied on the `/ws` route — middleware that mutates headers conflicts with the WS upgrade, and CORS isn't enforced by browsers on WebSocket handshakes.
